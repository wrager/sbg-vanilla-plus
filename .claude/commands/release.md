Выпустить релиз версии $ARGUMENTS (например: `/release 0.2.0`).

## Процесс

1. Определить предыдущий мажорный/минорный релиз (патч-фиксы не считаются — смотри на последний тег вида `vX.Y.0` или первый тег серии `vX.Y.Z`).
2. Проанализировать коммиты с момента предыдущего релиза (`git log <prev-tag>..HEAD --oneline`). Если предыдущего тега нет — все коммиты.
3. Сформировать `RELEASE_NOTES.md`:
   - Перечислить **новые фичи и изменения** с кратким описанием
   - **Фиксы** упоминать только если они исправляют баги **предыдущих версий** (а не фичи, добавляемые в этом же релизе — по ним дать итоговое описание)
   - Группировать по категориям (Стилизация / Фичи / Багфиксы) если применимо
4. Обновить `version` в `package.json` на указанную версию.
5. Запустить полную проверку: `npm run typecheck && npm run lint && npm run format:check && npm run test && npm run build`
6. Закоммитить все изменения (RELEASE_NOTES.md, package.json, и прочие если есть).
7. Создать тег `v<версия>`.
8. Запушить коммит и тег: `git push origin main --tags`.

Workflow `.github/workflows/release.yml` автоматически создаст GitHub Release с userscript в ассетах.
