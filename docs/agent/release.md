Выпустить релиз версии $ARGUMENTS (например: `/release 0.2.0`).

## Процесс

1. Определить предыдущий релиз — последний тег (`git describe --tags --abbrev=0`), независимо от того, patch это или minor/major.
2. Проанализировать коммиты с момента предыдущего релиза (`git log <prev-tag>..HEAD --oneline`). Если предыдущего тега нет — все коммиты.
3. Сформировать `RELEASE_NOTES.md`:
   - Начать с заголовка `# v{версия}` (например `# v0.2.0`)
   - Перечислить **новые фичи и изменения** с кратким описанием
   - **Фиксы** упоминать только если они исправляют баги **предыдущих версий** (а не фичи, добавляемые в этом же релизе — по ним дать итоговое описание)
   - Группировать по категориям (Стилизация / Фичи / Багфиксы) если применимо
4. Обновить `version` в `package.json` и `package-lock.json`: `npm version <версия> --no-git-tag-version`
5. Запустить полную проверку: `npm run typecheck && npm run lint && npm run format:check && npm run test && npm run build`
6. Закоммитить все изменения (RELEASE_NOTES.md, package.json, и прочие если есть).
7. Создать тег `v<версия>`.

**Не пушить.** Пользователь пушит сам. Workflow `.github/workflows/release.yml` автоматически создаст GitHub Release с userscript в ассетах.

**Релизный коммит** — только релизные файлы (RELEASE_NOTES.md, package.json, package-lock.json). Правки кода — отдельные коммиты ДО релиза.

## Правила составления release notes

- **Читать ПОЛНОЕ ТЕЛО каждого коммита.** Обязательный первый шаг — дамп `git log v<prev>..HEAD --format="=== %h %s ===%n%b%n" --reverse > /tmp/commits.txt` и прочитать файл целиком до написания первой фразы release notes. Заголовки и `--oneline`/`--stat` пропускают конкретику (квантификаторы, подфичи, состав изменений), которую автор зафиксировал именно в теле. Корневая ошибка: AI оптимизирует на минимум чтения через `--oneline` + README + grep, потому что чтение 400 строк git log кажется избыточным, — и систематически пропускает существенные детали.

- **Release notes = дельта v<prev>→v<new>, а не пересказ commit messages.** Commit message — это «что я изменил в этом коммите», release notes — это «что изменилось между v<prev> и v<new> с точки зрения пользователя». Для каждого пункта обязательно два вопроса: «когда эта функциональность появилась?» и «как я проверил формулировку?». Ответы «прочитал заголовок коммита» и «прочитал README модуля» недопустимы — первое берётся из `git log v<prev>..HEAD -- <файлы>`, второе из чтения текущего кода/тестов. Корневая ошибка: AI воспринимает задачу как литературную (переформулировать commit messages на пользовательский язык), а не аналитическую (описать дельту, проверенную против истории и кода).

- **Для ответа «что было в v<prev>» читать ПРОДУКТОВУЮ ЛОГИКУ, а не схемы настроек.** Наличие поля в типе настроек (`references: number`, `ReferencesMode`) не означает что функциональность активна: она может быть заглушкой (`addFlatDeletions(items, TYPE, -1)` с TODO-комментарием), фича-гейтом, мёртвым кодом. Для любого утверждения «раньше было X, теперь Y» открыть calculator/handler/executor (место реального исполнения), а не только файл настроек. Если в коде стоит константа/заглушка/TODO — фичи не было. Корневая ошибка: AI считает схему настроек достаточным описанием фичи, потому что её легче прочитать, чем код исполнения, и потому что в схеме есть словесные метки, которые легко пересказать.

- **Классификация «фикс предыдущей версии vs фикс фичи этого релиза».** Перед тем как положить коммит в секцию «Исправления», обязательно запустить `git log v<prev>..HEAD -- <файлы коммита>`: если в этом же диапазоне есть коммит, вводящий саму функциональность, — это фикс фичи ЭТОГО релиза, в «Исправления» не идёт, учитывается в описании фичи. Корневая ошибка: AI читает commit message и по его формулировке («после discover'а игра делает X, SVP делает Y») заключает о давнем взаимодействии, хотя сама интеграция была добавлена в этом же релизе.

- **Release notes описываются ПО МОДУЛЯМ.** Единица описания — модуль/фича, а не коммит. Для каждого модуля собрать все коммиты и описать итоговое изменение целиком. В пункте использовать имя модуля из таблицы в README (колонка с русским названием), а не внутренний id. Если коммит затрагивает интеграцию между модулями — описать в ОБОИХ модулях.

- **1 короткая фраза на фичу (~15–25 слов); детали — в README модуля.** Опускать: перечисления всех настроек/опций, технические детали реализации («через jQuery.localize», «автоматическая миграция»), объяснения «как это работает», обоснования «зачем это нужно», технические свойства основной функции («раздельные лимиты союзных/несоюзных» — свойство Slow-режима). НЕ опускать: независимо ценные пользовательские возможности — критерий «может ли пользователь использовать это отдельно от главной части фичи и имеет ли это собственный бенефит?». Пример: экспорт/импорт избранных в JSON — отдельный юзкейс (перенос между устройствами), упомянуть обязательно. Ориентир: секция ≈ 3–6 пунктов, всё release notes меньше страницы.

- **Квантификаторы не удалять при сокращении.** Слова «последний», «только», «всегда», «иногда», «при X», «для Y» определяют охват утверждения. Удаление превращает узкую защиту в общую: «защита **последнего** ключа от draw» → «не попадают в draw» — это ложь, draw работает как обычно, защищён только последний ключ. При каждом удалении слова задавать вопрос «если убрать, не расширится ли охват утверждения?». Если квантор не помещается в лимит длины — переформулировать так, чтобы он не понадобился, или убрать упоминание фичи вообще. Запрещено оставлять форму без квантора «потому что читатель поймёт» — не поймёт.

- **Русские термины — из `ILocalizedString.ru` и `docs/glossary.md`, а не из английского кода/commit messages.** Commit messages и код содержат английские идентификаторы (`references`, `Fast`, `Slow`, `References OPS tab`), и AI склонен копировать их напрямую, потому что они «уже готовы». Для любого видимого пользователю элемента UI (пункт меню, кнопка, вкладка, режим, название модуля, настройка) обязательно открыть `ILocalizedString` в `*Ui.ts`/`*SettingsUi.ts` и взять `.ru` вариант. Пример: `REF_SLOW_HINT.en = 'Slow cleanup runs manually from the references OPS tab'` → `.ru = 'Медленная очистка запускается вручную через кнопку во вкладке ключей в ОРПЦ'`. В `docs/glossary.md` зафиксированы русские эквиваленты (Ключ/Ref, ОРПЦ/OPS, СЛ, точка). Английские слова допустимы только для имён API и маркетинговых терминов без перевода.
