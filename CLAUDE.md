# Правила работы над проектом

Юзерскрипт для SBG (мобильная браузерная геолокационная игра). Модифицирует UI/UX.

## Критические запреты (нарушение = перманентный бан в игре)

1. **Запрещена подмена GPS**
2. **Запрещена автоматизация** игровых действий
3. **Запрещён мультиаккаунт**
4. **Запрещена модификация запросов** к серверу

## CI (перед каждым коммитом)

`npm run typecheck && npm run lint && npm run format:check && npm run test && npm run build`

Если `format:check` падает — запустить `npx prettier --write .`, а затем повторить.

## Git

- **ПЕРЕД КАЖДЫМ `git reset --soft` — обязательно `git log --oneline origin/<branch>..HEAD`.** Если 0 коммитов — reset запрещён (всё уже запушено). Если появились коммиты, которых не было раньше — origin обновился (кто-то запушил), пересчитать какие коммиты ещё не запушены. Корневая ошибка: AI запоминает «эти коммиты непушенные» из начала разговора и не перепроверяет перед каждой операцией с историей, хотя пользователь мог запушить между сообщениями
- **Коммит-сообщение описывает ТОЛЬКО то, что в коммите.** Если `git add` не включил файл (ошибка, вне репо, забыл), коммит-сообщение ОБЯЗАНО быть переписано с учётом фактического состава. Корневая ошибка: AI пишет коммит-сообщение заранее (до `git add`) по плану работы, а не по фактическому содержимому коммита. Если `git add` падает — AI повторяет коммит с тем же сообщением, не перечитывая его

## Код

- **Фикс бага = тест, воспроизводящий баг + тест на корректное поведение после фикса.** Если баг в том, что при disable показываются i18n-ключи вместо текста — тест должен мокнуть i18n-систему и проверить, что (1) при enable data-i18n убирается, (2) при disable data-i18n восстанавливается, (3) вызывается функция перевода. Тест, который проверяет только наличие атрибута, не покрывает суть фикса. Корневая ошибка: AI пишет тесты на «что я изменил в коде» (атрибут добавлен), а не на «какую проблему я решил» (текст должен быть переведён после disable)
- **При создании компактного/сводного UI из игровых элементов — переносить (reparent) оригинальные интерактивные элементы, а не копировать их текст в новые span'ы.** Копии теряют обработчики событий, data-атрибуты и игровую функциональность (клик по profile-link, hover и т.д.). Корневая ошибка: AI оптимизирует на «чистый код» (создание нового элемента вместо переноса DOM-ноды), игнорируя что оригинальный элемент несёт невидимое поведение (event listeners, jQuery delegation по классам)
- При изменении **lifecycle-кода** (init/enable/disable модулей) — проверять ВСЕ фазы, а не только изменяемую
- Перед реализацией новой фичи или перед сложным исправлением существующей — **сначала искать в `refs/`** (Read/Grep). Если `refs/` нет — запустить `npm run refs:fetch`
- **Не угадывать** DOM-классы/ID игровых элементов — посмотреть в референсах или запросить HTML из DevTools у пользователя
- **Не спрашивать у пользователя то, что есть в refs/.** Любые вопросы про игровые данные (типы предметов, API, DOM-структура) — сначала искать в `refs/`, только потом спрашивать. Корневая ошибка: AI считает быстрее спросить, чем искать — но пользователь не обязан быть справочником по собственному проекту
- Любое написание или изменение кода должно соответвовать docs/dev-principles.md и docs/codestyle.md
- Любое изменение в архитектуре или в ключевых механизмах должно соответстовать docs/architecture.md
- Любое использование терминов доменной области должно соответствовать словарю docs/glossary.md. Если используется новый термин, словарь должен актуализироваться в этом же коммите.
- **Перед определением нового типа/интерфейса инвентаря — проверить `src/core/inventoryTypes.ts` и `src/core/inventoryCache.ts`.** Типы предметов и функции чтения кэша централизованы в core. Дублирование запрещено
- **Перед реализацией утилит работы с OL Map — проверить `src/core/olMap.ts`.** Поиск слоёв (`findLayerByName`), управление DragPan (`createDragPanControl`) — централизованы
- **Перед чтением CSS custom properties — проверить `src/core/themeColors.ts`.** Цвета темы (`getTextColor`, `getBackgroundColor`) централизованы
- **Модули не должны импортировать друг друга напрямую.** Для межмодульного взаимодействия использовать `getModuleById()` из `moduleRegistry.ts`. Прямой `import` из `src/modules/X/` в `src/modules/Y/` запрещён
- **Async `install*()` должен защищаться флагом `pendingInstall`.** Если установка ресурса (observer, listener, layer) происходит после `await` (`waitForElement`, `getOlMap` и т.п.), синхронного guard'а `if (resource) return;` недостаточно — ресурс присваивается только в `.then()`, а второй вызов `install()` до резолва получит `resource === null` и пройдёт guard. Оба колбэка затем отвалятся по `generation !== installGeneration` (или установят ресурс дважды) — и модуль либо не инициализируется, либо утечёт. Нужно вводить `let pendingInstall = false;`, проверку `if (resource || pendingInstall) return;`, присвоение `pendingInstall = true;` перед запуском async-цепочки, сброс `= false` в `uninstall` и по завершении цепочки (success / catch). Корневая ошибка: AI считает synchronous-guard достаточным, опираясь на «успешное поведение при последовательных install→uninstall→install», и пропускает race-сценарий, который появляется только при async. `installGeneration` защищает от stale callbacks (установка observer'а в уже переустановленном модуле), но не от race — это разные проблемы.

## Исследование

- Перед новой фичей/правкой — **изучить модули проекта и референсы**
- **При изменении обработчиков UI-событий — исследовать ВСЕ связанные элементы в refs/.** Если код реагирует на кнопку X, проверить, нет ли вариантов этой кнопки (другие режимы, условия). Корневая ошибка: AI находит один элемент и считает задачу закрытой, не проверяя контекст — в результате пропускает аналогичные элементы
- При проблемах с API или платформой — **сначала документация/спецификация**, не перебирать варианты
- **`refs/` — единый канонический источник ЛЮБОГО чужого кода, с которым взаимодействует проект: игра, другие пользовательские скрипты (EUI/CUI), платформа/хост (SBG Scout), сторонние библиотеки.** Правило «сначала refs/» распространяется не только на игровые данные (DOM, API точек), но и на поведение любого внешнего кода, к которому проект имеет отношение. Перед тем как гадать, спрашивать пользователя или лезть в соседние директории типа `c:/projects/external/*-fix-my`, `-scout`, `-cui` — ОБЯЗАТЕЛЬНО `ls refs/` + `grep` по refs/. Если в refs/ нет — явно сообщить и предложить `npm run refs:fetch`, не искать альтернативы молча. Корневая ошибка: AI читает старое правило «сначала refs/» узко — как «про DOM-классы/API игры» — и при вопросе о стороннем userscript или хост-приложении **классифицирует его как «внешняя зависимость, не игра»** и не применяет правило. Вторая корневая ошибка: увидев в `ls` соседней директории «похожую по названию» папку, AI сразу идёт туда, потому что это первый ход с наименьшим усилием, пропуская шаг «а есть ли канонический источник в refs/?». Оба этих сокращения возникают из-за ложной экономии: кажется, что «где-то рядом в файловой системе» быстрее, чем «лишний ls в refs/», хотя разница в один инструмент-колл.

## Планирование

- **План изменений обязан содержать конкретный перечень тестовых кейсов, а не фразу «обновить тесты».** Для каждого изменения поведения в плане перечислить: (1) какие существующие тесты меняются и что именно в ассертах, (2) какие новые тесты добавляются, с описанием входа и ожидаемого выхода, (3) для логики с конечным числом состояний — полную таблицу истинности. Секция тестов в плане, помещающаяся в три строки, — маркер того, что покрытие не продумано. Корневая ошибка: AI рассматривает тесты как механическую доработку после реализации и в плане описывает их общими формулировками, экономя на том, что всё равно придётся додумать — из-за чего спецификация покрытия не обсуждается с пользователем заранее, и баги ловятся после коммита. Применять на этапе написания плана, до ExitPlanMode
- **Любой фидбек пользователя по результату предыдущей работы порождает в плане отдельную секцию «Улучшения правил».** В ней AI сам анализирует: какие пункты фидбека — это проявления систематических ошибок AI, и какие правила для CLAUDE.md предотвратят их повторение. Секция обязательна даже если фидбек кажется разовым — решение «разовый/системный» не принимается AI в одиночку. Для каждого пункта фидбека задать вопрос: «Если я завтра начну следующий модуль, сделаю ли я ту же ошибку?». Если ответ «возможно» или «не уверен» — это кандидат в правило. Правила формулировать через корневую поведенческую ошибку («AI делает X, потому что Y»), а не через механическую инструкцию («делай Z»). Корневая ошибка: AI воспринимает фидбек как точечную задачу «исправить конкретное место» и не делает шаг обобщения «какая моя привычка привела к этому», потому что точечное исправление быстрее и выглядит как завершённая работа — а превращение фидбека в правило требует саморефлексии, которая не входит в «минимальный путь до готового плана». Применять при получении любого фидбека на ранее сделанную работу, до ExitPlanMode

## Модули

- **Новые модули — `defaultEnabled: true`**, если пользователь явно не попросил иначе
- Создание нового модуля: команда `/new-module`
- **Чеклист перед коммитом модуля:** (1) тесты обновлены, (2) описание в README совпадает с `description.ru`, (3) название в README совпадает с `name.ru`
- **Каждый модуль с нетривиальной логикой — `README.md` в папке модуля.** Описывает назначение, возможности для игрока, архитектуру, защитные механизмы, настройки и файловую структуру. При изменении модуля — актуализировать README в том же коммите

## README

- Таблица модулей должна быть **всегда актуальна**: при изменении `name`, `id`, `category`, `description` — обновить в том же коммите
- **При любом изменении поведения — перечитать README целиком и проверить ВСЕ разделы на актуальность.** README содержит не только таблицу модулей: описание установки, способ доступа к настройкам, changelog, инструкции. Grep не достаточен — не все связи между изменениями и текстом README можно найти по ключевым словам. Коммит без проверки README = незавершённая работа. Корневая ошибка: AI рассматривает коммит как фиксированный чеклист «код + тесты + таблица модулей». Всё, что за пределами чеклиста (другие разделы README, документация), систематически пропускается, потому что AI не останавливается и не думает «что ещё затронуто моими изменениями?» — он выполняет известные шаги и считает работу завершённой
- **Порядок** = экран настроек: `CATEGORY_ORDER` (`ui` → `feature` → `map` → `utility` → `fix`), внутри категории — по порядку в `bootstrap()` в `entry.ts`

## Стили

- **Не хардкодить цвета** — CSS custom properties из refs/game/css/variables.css.
- Тема: `localStorage['settings']` → поле `theme` (`"light"` / `"dark"`)

## Внешние зависимости и refs

- **Если инструмент/fetch не обновил refs до ожидаемой версии — ОСТАНОВИТЬСЯ и сообщить пользователю.** Не переключаться молча на другой источник (release вместо source). Пользователь ожидает конкретную версию, и использование устаревших данных приведёт к ошибкам. Корневая ошибка: AI обнаруживает препятствие (fetch не обновил), обходит его молча и продолжает, вместо того чтобы эскалировать
- **Инфраструктурные правки (URLs, скрипты сборки, fetchRefs) — отдельный коммит от функциональных фиксов.** «Общий контекст» (оба про CUI) не делает правки одной темой. Корневая ошибка: AI спешит закончить и добавляет попутные правки в `git add` основного фикса, считая их «связанными»

## Релизы и пользовательские описания

- **Писать с точки зрения пользователя, не разработчика.** Описывать ЧТО изменилось для пользователя (какой баг исправлен, какая фича добавлена), а не КАК технически (какой механизм заменён, какой API использован). Не упоминать детали, являющиеся стандартным поведением (defaultEnabled и т.п.). Классифицировать изменения по фактическому содержанию коммита (багфикс или фича), а не по формулировке его заголовка.
- **Формулировать как действие, а не как описание проблемы.** Писать что сделано/исправлено («Исправлено X», «Добавлено Y»), а не что было сломано («X не работал», «Y отсутствовал»).
- **Релизный коммит — последний.** Вспомогательные коммиты (правила, документация) делать ДО релизного коммита.
- **При переделке — перезаписывать неудачные попытки.** Если незапушенные коммиты нужно переделать, использовать `git reset` для удаления неудачных попыток, а не наслаивать новые коммиты поверх.
- **Release notes: систематически пройти КАЖДЫЙ коммит.** При составлении release notes обязательно: (1) составить полный список коммитов и пройти по КАЖДОМУ — не пропускать «мелкие» фиксы, (2) прочитать diff или ключевые файлы, чтобы понять фактическое содержание, (3) прослеживать связи между коммитами (A сделано ДЛЯ B), (4) не включать в «Исправления» фиксы фич, добавленных в этом же релизе — по таким фичам давать итоговое описание. После составления — **кросс-проверка**: для КАЖДОГО пункта «Исправления» проверить, не относится ли он к модулю/фиче, описанному выше в «Новое»/«Улучшения». Если да — убрать из «Исправлений», информация уже учтена в итоговом описании фичи. Затем сверить список пунктов release notes со списком коммитов и убедиться, что каждый пользовательски значимый коммит учтён. Корневая ошибка: AI обрабатывает секции release notes (фичи, улучшения, фиксы) независимо — описывает модуль в «Улучшения», а потом тот же фикс этого же модуля дублирует в «Исправления», не связывая их.
- **Release notes: описывать изменения ПО МОДУЛЯМ, а не по коммитам.** Единица описания в release notes — модуль/фича, а не коммит. Алгоритм: (1) определить, какие модули затронуты коммитами, (2) для каждого модуля собрать ВСЕ коммиты и описать итоговое изменение модуля целиком, (3) если коммит затрагивает интеграцию между модулями — описать изменение в ОБОИХ модулях. Не пересказывать заголовки коммитов — читать diff и понимать фактическое содержание. Корневая ошибка: AI использует заголовки коммитов как источник текста для release notes, пересказывая их. Заголовок коммита — подсказка для поиска, а не готовый текст. Часть заголовка может казаться «вторичной» (после «и»), но для пользователя это отдельная значимая фича.
