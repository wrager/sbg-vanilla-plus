# SBG Vanilla+ — Правила AI-разработки

## Поведение AI

- AI **самостоятельно коммитит** изменения итеративно: правки по теме → проверка сборки → коммит → следующая тема
- **Один коммит = одна тема.** Несвязанные изменения — отдельные коммиты, даже если сделаны за один раз
- Перед каждым коммитом: `npm run typecheck && npm run lint && npm run format:check && npm run test && npm run build`
- Если `format:check` падает — сначала `npx prettier --write .`
- Все команды запускаются **без запроса разрешения**
- **Не делать `cd`** в Bash-командах — рабочая директория уже в корне проекта
- Коммиты на **русском языке**, кратко, всегда с trailerом `Co-Authored-By:` с именем текущей AI-модели
- Если правка относится к существующему коммиту (опечатка, уточнение, исправление ошибки **в рамках той же темы**) — вложить в тот коммит (**amend** для последнего, **fixup + rebase** для более старого), а не создавать новый коммит. Новая задача (даже если затрагивает тот же модуль) — всегда отдельный коммит. Ограничения: **не модифицировать старые коммиты** (только недавние, из текущей сессии); **запрещено амендить/fixup коммиты, которые уже запушены** — только новый коммит. Перед amend **обязательно** проверить `git status` на наличие расхождения с remote (`ahead/behind`): если текущий HEAD уже на remote — amend запрещён
- При замечании пользователя о работе AI — **немедленно обновить и этот файл, и memory** в том же ответе. Не одно из двух, а оба. Не откладывать, не спрашивать — сразу фиксировать правило, чтобы оно работало во всех будущих чатах. **Коммит с правилом — отдельный** от коммита с кодом (правило «один коммит = одна тема» важнее)
- **Увидел проблему — чини сразу.** Не отмечать и не спрашивать «завести задачу?» — если знаешь как починить, делай
- **Новые модули — `defaultEnabled: true`**, если пользователь явно не попросил иначе. Не додумывать «opt-in» — если фичу попросили, значит она нужна включённой
- При изменении кода — **актуализировать тесты, документацию и README**. **Тесты пишутся одновременно с кодом**, а не после: каждая новая ветка логики (ретрай, fallback, обработка edge case) сразу покрывается тестом в том же коммите. Нельзя написать код, а тесты «обновить потом»
- **Перед правкой существующей фичи** — изучить реализацию аналогичной функциональности в референсных скриптах (CUI, EUI), а не только код проекта
- **При проблемах с браузерными API** (touch events, pointer events, CSS свойства) — **первым делом изучить спецификацию W3C**, а не перебирать комбинации кода. Не угадывать поведение браузера, а знать его
- **При изменении lifecycle-кода** (init/enable/disable модулей) — проверять ВСЕ фазы, а не только изменяемую. Не потерять вызовы disable, не забыть обработку ошибок для каждой фазы
- **Коммиты с обоснованием**: в теле коммита фиксировать, почему выбран именно этот подход — чтобы история решений была доступна через `git log`
- **Планировать коммиты ДО кода.** Сначала определить, какие коммиты будут, потом писать код по одному коммиту за раз. Запрещено: написать всё сразу, а потом пытаться разделить на коммиты. **Конкретно:** нельзя редактировать файлы, относящиеся к разным коммитам, до того как первый коммит сделан. Цикл: определить коммиты → для каждого: правка **только его** файлов → проверка → коммит → следующий. При получении нескольких замечаний — тоже: определить коммиты, затем для каждого: правка → проверка → коммит → следующий. Не начинать все правки одновременно
- **Мат пользователя = системная ошибка AI.** Если пользователь матерится — AI допустил ошибку в подходе. Немедленно: (1) понять, какое правило нарушено или отсутствует, (2) создать/обновить правило в CLAUDE.md и memory, (3) только потом продолжить работу
- **При корректировке от пользователя — исправлять ВСЕ упоминания** в документе/плане/коде, а не только то место, на которое указали. Перед ответом — поиск по всему контексту на наличие устаревшей формулировки
- **При неоднозначной инструкции — переспросить**, а не угадывать. Лучше лишний вопрос, чем неправильная интерпретация
- Таблица модулей в README.md должна всегда отражать актуальное состояние: при любом изменении `name`, `id`, `category`, `description`, добавлении или удалении модуля — обновить таблицу в том же коммите. Название в колонке «Фича» совпадает с `name.ru`, описание — с `description.ru`
- **Порядок модулей в README.md** должен совпадать с порядком на экране настроек: группировка по `CATEGORY_ORDER` (`ui` → `map` → `feature` → `utility` → `fix`), внутри категории — по порядку в массиве `bootstrap()` в `entry.ts`
- **Релизный коммит — только релизные файлы** (RELEASE_NOTES.md, package.json, package-lock.json). Любые правки кода/тестов — отдельные коммиты ДО релиза. Версию обновлять через `npm version <версия> --no-git-tag-version` (обновит оба файла)
- **Правила RELEASE_NOTES.md:**
  - Предыдущий релиз = последний тег (включая patch). Не дублировать его фичи — сверяться с `git log <prev-tag>..HEAD` и предыдущим RELEASE_NOTES.md
  - Новые модули явно помечать как «Новый модуль: ...»
  - При упоминании модуля по ID — писать «модуль <название>», а не просто ID
  - Не-пользовательские изменения (рефакторинг категорий, внутренняя структура) — группировать в «Прочее» или убирать

## Критические запреты (нарушение = перманентный бан в игре)

1. **Запрещена подмена GPS**
2. **Запрещена автоматизация** игровых действий
3. **Запрещён мультиаккаунт**
4. **Запрещена модификация запросов** к серверу

Скрипт может ТОЛЬКО модифицировать UI/UX и **читать** (не изменять) ответы API.

## Добавление модуля

1. Создать `src/modules/<name>/<name>.ts` — экспортировать объект `IFeatureModule` (без side effects при импорте)
2. Добавить в `bootstrap([...])` в `entry.ts`
3. Добавить тесты в `src/modules/<name>/<name>.test.ts` (тесты лежат рядом с исходниками)

Детали архитектуры, стек → [docs/architecture.md](docs/architecture.md), конвенции именования → [docs/codestyle.md](docs/codestyle.md), словарь терминов → [docs/glossary.md](docs/glossary.md)

## Терминология

- **Использовать словарь** [docs/glossary.md](docs/glossary.md) при написании текстов (release notes, README, описания модулей). Не выдумывать расшифровки игровых терминов — сверяться со словарём
- В пользовательских текстах предпочитать русские варианты: **СЛ**, **ОРПЦ**, **точка**, **ключ**. Допустимы двуязычные: **СЛ/FW**, **ОРПЦ/OPS**
- **При исправлении пользователем термина** — немедленно обновить [docs/glossary.md](docs/glossary.md) (и memory). Словарь — источник истины для терминологии

## Типизация

- **Запрещён `as`-кастинг** без type guards — даже в тестах. Вместо `as` использовать type guards (`is`-функции, `in`-проверки, `instanceof`). В крайнем случае `as` допустим только с комментарием, объясняющим почему guard невозможен
- **Запрещено отключение ESLint** (`eslint-disable`, `eslint-disable-next-line` и т.д.) — даже в тестах. В крайнем случае допустимо только с коротким оправданием прямо в комментарии (например, `// eslint-disable-next-line @typescript-eslint/no-floating-promises -- fire-and-forget by design`)

## Игровые настройки и стили

- `localStorage['settings']` — JSON с настройками игры
  - Поле `theme` (`"light"` / `"dark"`) — текущая тема
- **Не хардкодить цвета** — использовать CSS custom properties игры из `:root`:
  - `--background`, `--background-transp` — фон
  - `--text`, `--text-disabled`, `--text-shadow` — текст
  - `--border`, `--border-transp` — рамки
  - `--accent`, `--selection`, `--link` — акценты
  - `--team-0..3`, `--level-1..10` — цвета команд и уровней
