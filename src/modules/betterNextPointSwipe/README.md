# betterNextPointSwipe

Заменяет нативный горизонтальный свайп игры на нашу логику переключения между точками с приоритетом по полезности.

## Поведение

Свайп влево или вправо на попапе точки (`.info`, вне карусели ядер) открывает следующую точку в радиусе действия игрока (45 м). Приоритет выбора:

1. Точки со свободными слотами для деплоя (`cores < 6`).
2. Точки, доступные для изучения (нет активного кулдауна).
3. Любая другая в радиусе.

Внутри каждого приоритета - ближайшая по расстоянию. Per-session трекинг посещённых: каждый свайп добавляет текущую точку в множество `visited`, следующий свайп её пропускает. Когда все точки в радиусе посещены - множество сбрасывается, цикл начинается заново.

## Различие с нативным поведением

Нативный горизонтальный свайп игры (`refs/game/script.js:722-752`):

- Циклически перебирает массив `near_points` (все точки в зоне видимости карты, в порядке как сервер отдал).
- Заполняется только при `map.on('click')` с фильтром `isInRange(f.p)` - если игрок не у точек, массив пустой.
- Проверка `if (near_points.length <= 1) return` блокирует свайп при пустом или единственном радиусе.

Наш модуль решает другую задачу: целенаправленный обход в зоне действия по приоритету полезности. Нативный handler подавляется - игрок переключается только по нашей логике.

## Архитектура

### Подавление нативного свайпа

Runtime-override на `Hammer.Manager.prototype.emit`. На событии `swipeleft`/`swiperight` с `target` внутри `.info` (но не внутри `.splide`-карусели ядер) наш handler выполняет навигацию и не вызывает оригинальный emit - нативный обработчик игры пропускается. На любом другом событии оригинальный emit вызывается как был.

Альтернатива - text-патч `hammer_info` в `gameScriptPatcher` (game-script модифицируется на загрузке) - не выбрана: runtime-override менее инвазивен, не требует обновления поисковой строки при минорных правках `script.js` игры. Подавление нативного срабатывает сразу при `enable` модуля и снимается при `disable`.

### Координация с nextPointSwipeAnimation

Если включён модуль `nextPointSwipeAnimation`, наш Hammer-override подавляет нативный handler, но НЕ вызывает свою навигацию - анимация сама выполнит её в своём `finalize` после dismiss-анимации. Проверка через `isModuleActive('nextPointSwipeAnimation')` из `core/moduleRegistry`. Если анимация выключена - навигация выполняется синхронно сразу при срабатывании emit.

### Открытие точки

Через `window.showInfo(guid)`, экспонированный из game-script через `core/gameScriptPatcher`. При недоступности `showInfo` (если patcher не отработал) - модуль no-op.

### Логика выбора

В `core/nextPointPicker.ts`. Чистые функции без модульного state, ctx с visited Set передаётся снаружи. Это позволяет переиспользовать логику в `nextPointSwipeAnimation`, каждый держит свой ctx.
