# Улучшенный свайп к следующей точке (improvedNextPointSwipe)

Заменяет нативный горизонтальный свайп между соседними точками на умную навигацию: только в радиусе действия игрока, с приоритетом по полезности.

## Возможности для игрока

- **Свайп влево/вправо в попапе точки - переход к следующей.** Жест тот же, что нативный, но логика выбора другая. Нативный свайп циклически перебирает все видимые на карте точки в порядке как сервер отдал; мы ходим только по точкам в радиусе действия (45 м от игрока) и в порядке полезности.
- **Приоритет по полезности.** Из множества кандидатов в радиусе модуль выбирает в порядке: (1) точки со свободными слотами для деплоя, (2) доступные для изучения (нет активного кулдауна), (3) любые. Внутри каждой группы - ближайшая по расстоянию.
- **Авто-зум при низком зуме.** Если в видимой области нет точек в радиусе, и текущий зум меньше 16, модуль кратковременно сдвигает viewport к игроку и зумит до 17. Игра автоматически подгружает точки в новой области, после чего попытка навигации повторяется. Старый viewport восстанавливается.
- **Per-session трекинг посещений.** Точка считается посещённой, когда её попап был открыт после нашего перехода. После reload счётчик обнуляется. Когда все in-range точки посещены - цепочка зацикливается с текущей.

## Зачем заменять нативное

Нативный свайп (refs/game/script.js:722-752) ходит по `near_points` - всем точкам в зоне видимости карты, в произвольном порядке. На большом зуме их сотни; в случае «у меня вокруг 5 точек, 3 пустых, хочу пройти их по очереди» нативный свайп бесполезен - он будет таскать в разные концы видимой области.

Наша логика делает «обход в радиусе действия по приоритету» - именно этот use case был у пользователя, когда он добавлял отдельную кнопку в более ранней версии модуля. Дальше выяснилось, что свайп - более удобный триггер, чем кнопка, и параллельная работа двух свайпов с разной семантикой запутывает: пользователь не помнит, чей конкретный свайп сейчас сработает. Поэтому модуль теперь полностью подменяет нативный, а не дополняет его.

## Архитектура

### Перехват через patch Hammer.Manager.prototype.emit

Игра вешает Hammer на `.info` для swipeleft/swiperight (`refs/game/script.js:722-752`) и его callback недоступен извне (закрыт в IIFE-замыкании). Прямого `manager.off()` сделать нельзя.

Модуль на `init` ставит patch на `Hammer.Manager.prototype.emit`. Каждый emit gesture-recognizer'a проходит через нашу функцию:

1. Если `interceptEnabled === false` - pass-through к `originalEmit`.
2. Если `name` - не `swipeleft` и не `swiperight` - pass-through.
3. Если `this.element` не `.info` - pass-through.
4. Иначе - вызываем `navigateInRange()` и НЕ зовём `originalEmit`. Нативный listener для этого события так и не получит управление.

Различение от других Hammer-instances на той же `.info` (модуль `swipeToClosePopup` тоже вешает Hammer для swipeup) идёт по name: vertical-recognizer не эмитит swipeleft/swiperight, и наоборот. Перехват достаточно узкий, чтобы не интерферировать.

Patch ставится один раз за жизнь страницы; повторные `installHammerInterceptor()` no-op. На `disable` patch остаётся, но при `interceptEnabled = false` вся ветка - pass-through, оверхед минимален.

### Навигация в радиусе

`tryNavigateInRange`:

1. Берёт текущий guid из `.info.popup[data-guid]`.
2. Добавляет его в `rangeVisited`.
3. Берёт координаты игрока из `player`-слоя.
4. Из всех features `points`-слоя оставляет те, что в `INTERACTION_RANGE = 45 м` от игрока.
5. Отфильтровывает уже посещённые в текущей цепочке.
6. Через `findNextByPriority` выбирает следующую: free-slots > discoverable > nearest.
7. Если все in-range посещены - очищает `rangeVisited`, оставляет только текущую и повторяет выбор. Получается циклический обход.
8. Открывает попап выбранной точки через `window.showInfo(guid)` (если `gameScriptPatcher` экспортировал её) или через симуляцию клика по координате.

### Авто-зум

Если в радиусе ничего нет (все точки далеко или ещё не подгружены), `autozoomAndNavigate`:

1. Сохраняет текущий center/zoom.
2. Устанавливает center = координаты игрока, zoom = 17.
3. Подписывается на `change` `points`-source: когда игра подгрузит новый набор точек после смены viewport, повторяем `tryNavigateInRange`.
4. По таймауту 3 сек или после первого `change` восстанавливает viewport.

### Geodesic distance

`getGeodeticDistance` использует `ol.sphere.getLength` на LineString из двух точек. Тот же метод, что игра в `isInRange`/`getDistance` (refs/game/script.js:2751-2757) - результат идентичен нативному критерию «точка в радиусе».

Внутри радиуса для упорядочивания по близости используется простое проецированное расстояние (`dx*dx + dy*dy`) - быстро и достаточно для сортировки в малом radius'е.

### Popup observer

`MutationObserver` на `.info` с фильтром `class` и `data-guid` - реагирует на смену открытой точки. Когда попап открывается с новым guid, обновляет `rangeVisited`. Если попап открылся для guid, отличного от того, к которому мы переходили (фейковый клик попал в соседнюю точку) - принимаем новый guid как часть цепочки.

Если первый fake-click не сработал (попап остался на старой точке), `MAX_FAKE_CLICK_RETRIES = 3` повторяет navigate перед отказом.

## Настройки

Модуль не имеет настроек. Включён по умолчанию.

## История

В предыдущей версии (`nextPointNavigation`) модуль работал параллельно с нативным свайпом и предоставлял отдельную кнопку `>` в попапе точки. После добавления этой кнопки в качестве UX выяснилось, что параллельный свайп нативный + кнопка наш - запутывает (пользователь не помнит, чья навигация сейчас активна), и более удобно полностью заменить нативный свайп нашим. Кнопка убрана, id переименован, модуль теперь активирует тот же жест, что и нативный, но по своей логике.
