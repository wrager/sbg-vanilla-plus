# Автоочистка инвентаря (inventoryCleanup)

Модуль автоматически удаляет лишние предметы из инвентаря перед изучением точки. Поддерживает два режима удаления ключей: быстрый (автоматический) и медленный (ручной с определением фракции).

## Адаптация к SBG 0.6.1+

В SBG 0.6.1 появился нативный «Сборщик мусора» с серверной чисткой инвентаря по лимитам. Модуль адаптирован: пока он активен, нативный сборщик принудительно отключается через `nativeGarbageGuard` (POST `/api/settings { usegrb: false }` плюс DOM-disable инпутов «Сборщика мусора» в нативных настройках), и активной остаётся только наша автоочистка. Защита ключей перешла на нативный lock-флаг (бит 0b10 поля `f` стопки в `inventory-cache`), при отсутствии lock-поддержки в кэше удаление ключей блокируется. Дополнительно: пока локальный список SVP/CUI-избранных не перенесён в нативные замочки через `favoritesMigration`, удаление ключей блокируется до завершения миграции.

## Для игрока

### Когда срабатывает

Очистка запускается **после завершения изучения точки** — когда сервер ответил на запрос и игра обновила инвентарь. Срабатывает для всех трёх кнопок изучения: обычное изучение, изучение без ключа и изучение только ключа. Если кнопка неактивна (точка вне зоны действия, кулдаун) — очистка **не** срабатывает. При простановке (Deploy) очистка также **не** срабатывает.

### Условие запуска

Очистка происходит, только если в инвентаре **мало свободных слотов**. Порог задаётся в настройках параметром «Мин. свободных слотов» (по умолчанию 100).

### Что удаляется

| Тип предмета       | Как задаётся лимит                                             | Значение -1     |
| ------------------ | -------------------------------------------------------------- | --------------- |
| Ядра (Cores)       | Отдельный лимит на уровень                                     | Без ограничений |
| Катализаторы       | Отдельный лимит на уровень                                     | Без ограничений |
| Ключи (References) | Лимит на точку, защита по нативному `lock` (бит 0b10 поля `f`) | Без ограничений |

Мётлы и специальные предметы никогда не удаляются.

По умолчанию все лимиты установлены в -1 (без ограничений), так что без настройки ничего не удаляется.

### Режимы удаления ключей

- **Off** (по умолчанию) — ключи не удаляются
- **Fast** — автоматическое удаление при discover. Лимит на точку, без разделения по фракции. Защита: ключи locked-точек (бит 0b10 поля `f` в `inventory-cache`) не удаляются. Если в кэше нет ни одной стопки с полем `f` (старые версии SBG без lock-семантики) — удаление ключей блокируется. Если у пользователя остался legacy SVP/CUI-список избранного и активен модуль `favoritesMigration` — удаление ключей блокируется до завершения миграции
- **Slow** — ручное удаление по кнопке «Очистить ключи» во вкладке ключей инвентаря. Запрашивает `/api/point` для определения фракции каждой точки, затем применяет раздельные лимиты для союзных и несоюзных

### Порядок удаления (FIFO)

Когда количество предметов превышает лимит, удаляются предметы **с начала инвентаря** — те, что были получены раньше.

### Уведомления

```
Очистка (13): Я5 ×10, К3 ×3   (русский)
Очистка (13): Co5 ×10, Ca3 ×3 (английский)
```

### Настройки

Настройки доступны через кнопку «Настроить» рядом с модулем. Экран настроек позволяет задать:

- **Мин. свободных слотов** — при каком количестве свободных слотов запускать очистку
- **Ядра** — лимит для каждого уровня (1–10)
- **Катализаторы** — лимит для каждого уровня (1–10)
- **Ключи** — режим (off/fast/slow), лимит на точку для fast, лимиты союзных/несоюзных для slow

## Техническая реализация

### Источники данных

- **Количество предметов и лимит инвентаря** — DOM: `#self-info__inv`, `#self-info__inv-lim`
- **Содержимое инвентаря** — `localStorage['inventory-cache']`, JSON-массив:
  - `t=1` — ядра (`l` = числовой уровень)
  - `t=2` — катализаторы (`l` = числовой уровень)
  - `t=3` — ключи (`l` = строковый GUID точки)
  - `t=4` — мётлы (игнорируются)

### Алгоритм (fast mode)

1. Клик по кнопке discover → флаг `discoverPending`
2. Перехватчик `localStorage.setItem` ждёт записи в `inventory-cache`
3. Проверка: `inventoryLimit - currentCount < minFreeSlots`
4. `calculateDeletions`: группировка по уровню/точке, определение излишка, FIFO
5. Ключи проходят guard: mode=fast, limit≠-1, lockSupportAvailable (хотя бы одна стопка с полем `f`); locked-точки (бит 0b10) исключены из расчёта; если активен `favoritesMigration` и legacy IDB непуст — режим ключей принудительно `off`
6. `deleteInventoryItems`: финальный guard с перечитыванием свежего `inventory-cache` и проверкой каждой точки на актуальный lock-бит
7. Тост с итогом

### Алгоритм (slow mode)

1. Пользователь нажимает кнопку «Очистить ключи» в инвентаре
2. Проверка lock-поддержки: в `inventory-cache` есть хотя бы одна стопка с полем `f`. Кнопка скрыта, пока legacy SVP/CUI-список непуст и модуль `favoritesMigration` активен
3. Фильтрация: ключи locked-точек (бит 0b10) исключены из расчёта
4. Запрос `/api/point` для каждой уникальной точки (concurrency = 3)
5. Группировка по фракции (allied / notAllied), применение раздельных лимитов
6. `deleteInventoryItems` с финальным lock-guard
