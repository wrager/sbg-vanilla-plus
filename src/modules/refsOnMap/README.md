# Ключи на карте (refsOnMap)

Просмотр и управление ключами в инвентаре через визуализацию точек на карте на любом масштабе.

## Для игрока

### Запуск

Кнопка «На карте» в инвентаре, на вкладке ключей. Кнопка вставляется рядом с нативной `#inventory-sort`.

### Что делает

- Открывает full-screen viewer поверх игровой карты: подгружает все ключи из `inventory-cache`, для каждой стопки рисует кружок в координатах точки. Один кружок = одна стопка ключей (несколько стопок одной точки рисуются отдельно).
- Кружок раскрашен цветом команды точки. Команды грузятся двумя путями параллельно: (a) active pull через `/api/point` для видимых в текущем extent ref-точек при открытии viewer и на каждый `moveend` карты; (b) enrichment через перехват игрового `/api/inview` — когда игра дёргает этот эндпоинт сама, hook ловит ответ и сразу пишет команды в кэш, минуя `/api/point`.
- Клик по кружку выбирает/снимает выбор. Внизу справа появляется кнопка «Корзина» с количеством выбранных ключей. Клик по корзине удаляет выбранные стопки через DELETE `/api/inventory`.
- Закрытие — кнопка `[x]` снизу.

### Защита от удаления

- **Замочек или звёздочка (нативный SBG 0.6.1)** — ключи защищённых точек игнорируются при удалении даже если выбраны. После удаления показывается toast «Locked-точки: N ключей оставлено».
- **«Не удалять свои» (опциональный чекбокс)** — когда выбрана хотя бы одна точка, справа в viewer появляется label с чекбоксом. Когда включён: при удалении из payload исключаются ключи точек, чья команда совпадает с командой игрока. После удаления toast «Свои: N ключей оставлено». **Состояние эфемерное** — действует только до закрытия viewer, в `localStorage` не сохраняется.

### Прогресс-бар и блокировки

Active pull через `/api/point` worker батчами по 5 (delay 100 мс между батчами) запускается при `showViewer` и на каждом `moveend` — добавляет в очередь все видимые ref-точки, чьих команд нет в кэше. `/inview`-перехват параллельно ловит игровые ответы; если игра уже принесла `t` для guid'а, который ещё в очереди, hook удаляет его из очереди и worker пропускает.

Пока в очереди что-то есть:

- Видим прогресс-бар сверху по центру (счётчик `done / total`).
- **Кнопка «Корзина» disabled.**
- **Клик по карте не выбирает фичу** (фильтр «не удалять свои» опирается на финальные значения команды у каждой фичи; выбор до загрузки привёл бы к несогласованному payload).
- Pan и zoom карты работают как обычно — но дополнительный `moveend` добавит в очередь новые видимые точки, и блокировки продлятся.

После выработки очереди прогресс-бар скрывается, кнопки и клики активны. На последующий pan в новую область прогресс-бар появится снова, если там есть незагруженные точки.

### Чекбокс «Не удалять свои»

- **Видимость:** скрыт пока выбора нет (`uniqueRefsToDelete === 0`); появляется при первом выборе фичи; скрывается обратно после снятия всех выборов.
- **State эфемерный:** при открытии viewer чекбокс всегда выключен, независимо от прошлого сеанса. `localStorage` не используется.
- **Auto-сброс на смену видимой области:** появление новых (ранее не виденных) guid'ов в visible extent при включённом чекбоксе **автоматически выключает** фильтр и показывает toast «Фильтр 'Не удалять свои' сброшен из-за возможного появления точек с неизвестной командой». Триггеры два: active pull на `moveend` (новые точки в visible extent после pan'а) и `/api/inview`-перехват (игра принесла новые guid'ы раньше нашего pull'а). Корневой принцип: новый набор видимых точек = новый контекст, прежний фильтр невалиден. Сброс срабатывает даже если новые guid'ы не в текущем selection.

### Поведение при `playerTeam = null`

Команда игрока определяется через `var(--team-N)` в `style.color` элемента `#self-info__name` (источник — игра, см. `src/core/playerTeam.ts`). Если команду извлечь не удалось:

- При выключенном чекбоксе «Не удалять свои» — обычное удаление без фильтра.
- При включённом — удаление **заблокировано** с toast «Не удалось определить команду игрока». Пользователь должен либо выключить фильтр, либо разобраться почему команда не определилась.

Это симметрично с поведением `slowRefsDelete` в `inventoryCleanup`: явный пользовательский фильтр без возможности его выполнить требует подтверждения, а не молчаливого пропуска.

### Поведение при `team === undefined` у фичи (fail-safe)

При включённом чекбоксе «Не удалять свои», если у фичи свойство `team` не установлено (active pull ещё не завершился, `/api/point` упал, точка ни в visible extent при `showViewer`, ни во `moveend`-сессиях) — фича попадает в отдельный bucket `protectedByUnknownTeam` и не удаляется.

`protectedByUnknownTeam` отделён от `protectedByOwnTeam`:

- «Свои оставлены» — поведение по дизайну, ничего делать не надо.
- «Цвет не загружен» — техническая проблема, в тосте подсказка pan'нуть карту или переоткрыть viewer.

## Файловая структура

| Файл                | Назначение                                                                      |
| ------------------- | ------------------------------------------------------------------------------- |
| `refsOnMap.ts`      | Модуль: lifecycle, viewer-state, `/inview`-перехват, рендеринг, удаление        |
| `styles.css`        | Стили кнопок, locked-note, прогресс-бара, чекбокса «Не удалять свои»            |
| `refsOnMap.test.ts` | Тесты: lifecycle, viewer, lock, own-team, `/inview`-driven load, checkbox, lock |

## Связанные модули

- `inventoryCleanup` — общая семантика lock/favorite-защиты. Та же функция `buildLockedPointGuids` из `core/inventoryCache.ts` используется для определения защищённых точек.
- `core/playerTeam` — вынесена общая `getPlayerTeam()`, переиспользуется со `slowRefsDelete`.
- `core/refsHighlightSync` — после успешного удаления ключей вызывается `syncRefsCountForPoints` для актуализации подписи на основной карте.
- `refsLayerSync` — использует тот же паттерн monkey-patch `window.fetch` (для `/api/discover`); refsOnMap перехватывает `/api/inview`, цепочка hook'ов не конфликтует.
