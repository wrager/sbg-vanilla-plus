# Ключи на карте (refsOnMap)

Просмотр и управление ключами в инвентаре через визуализацию точек на карте на любом масштабе.

## Для игрока

### Запуск

Кнопка «На карте» в инвентаре, на вкладке ключей. Кнопка вставляется рядом с нативной `#inventory-sort`.

### Что делает

- Открывает full-screen viewer поверх игровой карты: подгружает все ключи из `inventory-cache`, для каждой стопки рисует кружок в координатах точки. Один кружок = одна стопка ключей (несколько стопок одной точки рисуются отдельно).
- Кружок раскрашен цветом команды точки. Команды грузятся двумя путями параллельно: (a) active pull через `/api/point` для видимых в текущем extent ref-точек при открытии viewer и на каждый `moveend` карты; (b) enrichment через перехват игрового `/api/inview` — когда игра дёргает этот эндпоинт сама, hook ловит ответ и сразу пишет команды в кэш, минуя `/api/point`. Worker делает double-check `teamCache.has` перед каждым `/api/point` — если /inview успел положить команду между извлечением guid'а из очереди и реальным запросом, лишний fetch не отправляется.
- Клик по кружку выбирает/снимает выбор. Справа внизу появляется кнопка «Корзина» с числами `X (X ключей)` — точек и ключей, готовых к удалению, и кнопка «Отменить» под ней (снимает выделение со всех выбранных точек). Справа над «Корзиной» — блок-сводка по выбору и опциональный чекбокс «Не удалять свои» (см. ниже). Клик по корзине удаляет deletable-стопки через DELETE `/api/inventory`.
- Закрытие — кнопка `[x]` снизу.

### Сводка по выбору

Блок справа внизу, виден когда выбрана хотя бы одна стопка. До шести строк (lock/own/unknown/keepone — опциональные, видны только когда соответствующий bucket не пуст):

- **«Всего: X (X ключей). Из них:»** — суммарный выбор без учёта защит. Точки считаются по уникальным `pointGuid` (две стопки одной точки = 1 точка).
- **«X (X ключей) защищено»** — точки с lock (замочек/звёздочка). Опциональная, видна когда lock-bucket не пуст.
- **«X (X ключей) своего цвета»** — точки команды игрока. Опциональная, видна ТОЛЬКО при включённом чекбоксе «Не удалять свои» и когда own-bucket не пуст.
- **«X (X ключей) неизвестного цвета»** — выбранные точки, для которых `team` не удалось загрузить (fetch упал, ответ нераспознан). Опциональная, видна ТОЛЬКО при включённом чекбоксе «Не удалять свои» и когда unknown-bucket не пуст. Нейтральные точки (`data.te: null` в ответе сервера) сюда НЕ попадают — у них `feature.team === null`, они идут в deletable.
- **«X (X ключей) защищено (правило "1 ключ")»** — точки, которые keepOneKey-фильтр полностью защитил (типичный сценарий: точка с 1 ключом, выделена). Опциональная, видна когда чекбокс «Оставлять 1 ключ» включён и есть такие точки. Частично-удалённые точки (где удаляется amount-1, остаётся 1 ключ) сюда не попадают — они в deletable с уменьшенным amount.
- **«X (X ключей) к удалению»** — то же число, что на кнопке «Корзина», продублировано в блоке для удобства чтения сводки слева направо. Учитывает keepOneKey: показывает фактическую сумму payload.

Bucket'ы `lock` / `own` / `unknown` / `keepOneKey` / `deletable` disjoint по построению `partitionByLockProtection` + `computeDeletablePayload`. Сумма точек по этим строкам = «Всего».

На кнопке «Корзина» — только deletable: `X (X ключей)`. При защите всего выбора (lock/own/unknown/keepOneKey) кнопка покажет `0 (0 ключей)`, клик по ней покажет тост-разъяснение и не уйдёт в DELETE.

Все числа пересчитываются: (1) при каждом клик-toggle по фиче; (2) при переключении чекбокса «Не удалять свои» или «Оставлять 1 ключ»; (3) после каждого batch'а worker загрузки команд; (4) после успешного DELETE.

### Защита от удаления

- **Замочек или звёздочка (нативный SBG 0.6.1)** — ключи защищённых точек игнорируются при удалении даже если выбраны. lock-семантика per-point: одна locked-стопка защищает ВСЕ стопки точки (агрегация в `buildLockedPointGuids`). После удаления показывается toast «Locked-точки: N ключей оставлено».
- **«Не удалять свои» (опциональный чекбокс)** — когда выбрана хотя бы одна точка, справа в viewer появляется label с чекбоксом. Когда включён: при удалении из payload исключаются ключи точек, чья команда совпадает с командой игрока. После удаления toast «Свои: N ключей оставлено». **Состояние персистентное** — сохраняется в `localStorage['svp_refsOnMap'].keepOwnTeam`.
- **«Оставлять 1 ключ» (чекбокс, дефолт включён)** — гарантирует, что после DELETE у каждой выделенной точки в инвентаре останется минимум 1 ключ. Учитываются и невыделенные стопки этой точки: если они уже дают >=1 ключ, выделенные удаляются полностью; иначе из выделенных стопок distribute удаляется `selected_total - 1` ключей. Точки, где `selected_total <= 1`, полностью защищены (попадают в keepOneKey-bucket). Состояние персистентное (`localStorage['svp_refsOnMap'].keepOneKey`); fail-safe: при отсутствии или повреждении записи флаг считается true.

Инвариант, проверяемый отдельной серией critical-safety тестов:

- guid из protected-bucket (lock/own/unknown) НЕ может попасть в payload DELETE ни при одной комбинации.
- При keepOneKey=true сумма payload по точке СТРОГО МЕНЬШЕ суммарного inventory amount этой точки. То есть после DELETE у каждой выделенной точки в инвентаре остаётся минимум 1 ключ.

Регрессия здесь = пользователь теряет защищённые ключи (или все ключи точки при keepOneKey) без своего ведома.

### Прогресс-бар и блокировки

Active pull через `/api/point` worker батчами по 5 (delay 100 мс между батчами) запускается при `showViewer` и на каждом `moveend` — добавляет в очередь все видимые ref-точки, чьих команд нет в кэше. `/inview`-перехват параллельно ловит игровые ответы.

Пока в очереди что-то есть:

- Видим прогресс-бар сверху по центру (счётчик `done / total`).
- **Кнопка «Корзина» disabled — ТОЛЬКО когда включён чекбокс «Не удалять свои» И хотя бы одна выбранная точка реально в очереди загрузки команды (`teamLoadQueue` или `teamLoadInFlight`).** Фоновая загрузка команд для невыбранных точек (`moveend`, `/inview enrichment`) кнопку не блокирует: фильтр свои читает `team` только у selected, а lock защищается через `inventory-cache.f`. Проверка — `hasSelectedPointsLoadingTeam()`, реактивно пересчитывается из `updateSelectionUi` на toggle, на change чекбокса и после каждого batch'а worker'а.
- **Клик по карте работает всегда.** При keepOwnTeam=true выбранная точка с team=undefined попадёт в `protectedByUnknownTeam` (fail-safe), удаление её защитит.
- Pan и zoom карты работают как обычно — `moveend` добавит в очередь новые видимые точки, прогресс-бар продлится, но если ни одна из них не выбрана пользователем, кнопка «Корзина» останется доступной.

После выработки очереди прогресс-бар скрывается. На последующий pan в новую область прогресс-бар появится снова, если там есть незагруженные точки.

### Текст чекбокса keepOwnTeam под цвет команды игрока

Чекбокс «Не удалять свои» в реальном UI показывается как «Не удалять красные» / «Не удалять зелёные» / «Не удалять синие» в зависимости от `getPlayerTeam()` (`--team-1`/`--team-2`/`--team-3` в `refs/game/css/variables.css`). Если команда игрока не извлекается (`#self-info__name` пустой / другой формат / выход из аккаунта) — fallback на исторический «Не удалять свои» / «Keep own team». Текст перечитывается при каждом `showViewer`, чтобы переключения аккаунта внутри одной страницы корректно отражались.

Везде ниже в этом README пишем «Не удалять свои» как обобщающее имя чекбокса.

### Чекбоксы «Не удалять свои» и «Оставлять 1 ключ»

- **Видимость:** оба скрыты пока выбора нет; появляются при первом выборе фичи; скрываются обратно после снятия всех выборов.
- **State персистентный:** оба значения сохраняются в `localStorage['svp_refsOnMap']` через `refsOnMapSettings.ts` (`load/save`). `showViewer` восстанавливает чекбоксы из storage, `onChange` каждого сохраняет оба поля.
- **Дефолты:** `keepOwnTeam=false` (opt-in, исторически viewer удалял всех), `keepOneKey=true` (fail-safe, защита нового пользователя от случайного полного удаления ключей точки). При повреждении или отсутствии записи в localStorage оба возвращаются к дефолтам пер-полно.
- **Не сбрасывается на смену видимой области.** Точки без `team` (ещё не загружены) защищены через `protectedByUnknownTeam` fail-safe в `partitionByLockProtection`. keepOneKey не зависит от загрузки team и работает всегда.

### Алгоритм keepOneKey (computeDeletablePayload)

Применяется к каждой точке в deletable-bucket (после lock/own/unknown-фильтрации):

1. Прочитать все стопки этой точки из `inventory-cache` (включая невыделенные и lock-стопки).
2. `selectedAmount = sum amount выделенных стопок этой точки`.
3. `inventoryTotal = sum amount всех стопок` точки.
4. `unselectedAmount = inventoryTotal - selectedAmount`.
5. Если `unselectedAmount >= 1` или `keepOneKey=false` — payload получает все выделенные стопки целиком.
6. Иначе (`unselectedAmount === 0`, нет невыделенных стопок):
   - `toDeleteTotal = selectedAmount - 1`.
   - Если `toDeleteTotal <= 0` — все стопки точки уходят в `protectedByKeepOneKey` (нечего удалять без нарушения инварианта).
   - Иначе distribute `toDeleteTotal` среди стопок по убыванию `amount` (тай-брейк по `refGuid` ASC для детерминированности). Стопки с `delete_amount > 0` уходят в payload; стопки с `delete_amount = 0` — в `protectedByKeepOneKey`.

После DELETE: полностью удалённые стопки убираются из `refsSource` и из `inventory-cache`; частично-удалённые обновляют `amount` и сбрасывают `isSelected` (визуально точка-кружок становится не выбранным).

### Поведение при `playerTeam = null`

Команда игрока определяется через `var(--team-N)` в `style.color` элемента `#self-info__name` (источник — игра, см. `src/core/playerTeam.ts`). Если команду извлечь не удалось:

- При выключенном чекбоксе «Не удалять свои» — обычное удаление без фильтра.
- При включённом — удаление **заблокировано** с toast «Не удалось определить команду игрока». Дополнительно `computeSelectionBreakdown` defensive-режимом считает ВСЁ selected как `protectedByUnknownTeam`: UI показывает «0 (0 ключей)» на кнопке и весь выбор как «защищено». Это устраняет inconsistency «UI говорит N deletable, но кнопка не удалит» и закрывает класс ошибок, при котором падение guard в `handleDeleteClick` пропустит чужие/свои/unknown в payload.

Это симметрично с поведением `slowRefsDelete` в `inventoryCleanup`: явный пользовательский фильтр без возможности его выполнить требует подтверждения, а не молчаливого пропуска.

### Значения `feature.team` и связь с bucket'ами

- `number` — конкретная команда (1..4). При `keepOwnTeam=true` сравнивается с `playerTeam`: совпадение → `protectedByOwnTeam`, иначе → `deletable`.
- `null` — нейтральная точка. Сервер ответил 200 OK с `data.te: null`, точка без владельца — легитимное игровое состояние. `feature.team` явно ставится в `null`, точка идёт в `deletable` (не своя). До разделения neutral и failed склеивались, и нейтральные точки попадали в protected — пользователь видел «X неизвестного цвета» после полной отработки worker'а.
- `undefined` — команда не загружена. Active pull ещё не завершился, точка не в visible extent при `showViewer`/`moveend`, либо `fetchPointTeam` вернул `'failed'` (fetch упал, ответ нераспознан). При `keepOwnTeam=true` точка попадает в `protectedByUnknownTeam` fail-safe и не удаляется.

`protectedByUnknownTeam` отделён от `protectedByOwnTeam`:

- «Свои оставлены» — поведение по дизайну, ничего делать не надо.
- «Цвет не загружен» — техническая проблема, в тосте подсказка pan'нуть карту или переоткрыть viewer.

### Кэш команд `teamCache`

`Map<pointGuid, number | null | 'failed'>`. Три значения соответствуют трём вариантам `feature.team` выше плюс sentinel `'failed'` (не пишется в `feature.team`, оставляет `undefined`). `teamCache.has(guid)` skip'ит и успешные, и нейтральные, и провалившиеся попытки — иначе каждый `moveend` через extent заново ставил бы guid в очередь, worker дёргал бы `/api/point` бесконечно.

## Связанные модули

- `inventoryCleanup` — общая семантика lock/favorite-защиты. Та же функция `buildLockedPointGuids` из `core/inventoryCache.ts` используется для определения защищённых точек.
- `core/playerTeam` — вынесена общая `getPlayerTeam()`, переиспользуется со `slowRefsDelete`.
- `core/refsHighlightSync` — после успешного удаления ключей вызывается `syncRefsCountForPoints` для актуализации подписи на основной карте.
- `refsLayerSync` — использует тот же паттерн monkey-patch `window.fetch` (для `/api/discover`); refsOnMap перехватывает `/api/inview`, цепочка hook'ов не конфликтует.
