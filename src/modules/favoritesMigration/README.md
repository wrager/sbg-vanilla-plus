# Миграция избранного (favoritesMigration)

Однократная миграция локального списка избранных точек SVP/CUI в нативные «звёздочки» и «замочки» инвентаря SBG 0.6.1+.

## Возможности для игрока

В настройках модуля доступны:

- **⬆️ Импорт из JSON** — загружает локальный список SVP/CUI-избранных из ранее экспортированного JSON. **REPLACE-семантика**: текущий список в IDB полностью заменяется. Используется при переносе на новое устройство или восстановлении после wipe. После импорта флаг `lock-migration-done` сбрасывается — свежие легаси-точки требуют повторной миграции в нативные замочки.
- **⬇️ Скачать JSON** — выгружает текущий список в файл `svp-favorites-YYYY-MM-DD.json`. Формат — массив GUID-строк, отсортированный.
- **Сделать их избранными** — для каждой точки из локального списка SVP/CUI с ключами в инвентаре отправляется `POST /api/marks { guid, flag: 'favorite' }`. Это нативная «звёздочка» SBG 0.6.1: визуальное маркирование, не защищает от удаления автоочисткой.
- **Сделать их заблокированными** — то же, но с `flag: 'locked'`. Нативный замочек: ключи не участвуют в рисовании и не удаляются автоочисткой `inventoryCleanup`. После успешной миграции выставляется флаг `svp_lock_migration_done`, и `inventoryCleanup` перестаёт блокировать удаление ключей по факту непустого legacy-списка.

После миграции локальный список SVP/CUI **не очищается**: миграцию можно повторить при необходимости (например, после случайного снятия флага в инвентаре игры).

## Поведение

### Per-stack обработка

Игра в SBG 0.6.1 хранит ключи как стопки (`item.t === 3`, `item.l === pointGuid`, `item.g === stackGuid`). У одной точки может быть несколько стопок инвентаря (см. `release-notes 4.E`). Миграция вызывает `POST /api/marks` для **каждой стопки** избранной точки — флаг применяется на уровне стопки.

В UI инвентаря игры стопки агрегируются в одну строку, поэтому пользователь видит «звёздочка стоит у этой точки» независимо от того, на скольких стопках она физически отмечена.

### Concurrency

Параллельных запросов одновременно — не более 4 (паттерн `scheduleLimitedPointFetch`). Защищает сервер от залпа при списке из десятков избранных.

### Retry-механизм

Запросы `POST /api/marks` могут провалиться двумя способами:

1. **Сетевая ошибка** (fetch reject, HTTP не 2xx) — пользователю показывается `confirm('Не удалось применить флаг для N стопок. Повторить?')`. На «ОК» список упавших стопок повторяется. Цикл продолжается, пока: (а) все успешны, или (б) пользователь нажал «Отмена».
2. **Toggle off** (HTTP 200, но `response.result === false`) — означает, что у стопки уже стоял нужный флаг и API toggle'нул его OFF (наш предварительный фильтр не сработал из-за устаревшего `f` в кэше). Один автоматический retry: повторный вызов вернёт флаг в ON. Если и второй раз false — стопка идёт в `toggleStuck`, в финальном toast с описанием.

## Хранилище

Source данных миграции — IndexedDB `CUI/favorites` (тот же store, что у CUI). На init модуль вызывает `loadFavorites()` из `core/favoritesStore`, который заполняет in-memory `Set<string>` GUID'ов точек.

После переименования из `favoritedPoints` в `favoritesMigration` точечные write-методы (addFavorite, removeFavorite) удалены: модуль не редактирует список по-точкам, нативные звёздочки/замочки игра пишет в свой формат `inventory-cache[i].f`. REPLACE-методы `exportFavoritesToJson` / `importFavoritesFromJson` сохранены в `core/favoritesStore` — нужны для переноса легаси-списка между устройствами и восстановления после wipe.

### Count seal (защита от потери IDB)

Сохранён из старой реализации: если IDB пуста, а localStorage `svp_favorites_seal > 0` — значит, данные потеряны (Android IDB wipe). `isFavoritesSnapshotReady()` возвращает false, `inventoryCleanup` не удаляет ключи, пользователь получает alert.

## Эндпоинт `POST /api/marks`

Описан в [`release-notes 4.A`](../../../C:/tmp/sbg-beta-release-notes.md). Payload `{ guid: string, flag: 'favorite' \| 'locked' }`. Ответ `{ response: { result: boolean } }` — `result === true` означает, что флаг УСТАНОВЛЕН после toggle, `false` — снят.

Игровой `apiSend` — IIFE-внутренняя функция и недоступна юзерскрипту. Используется прямой `fetch` с auth-токеном из `localStorage['auth']` (паттерн из `inventoryApi.deleteInventoryItems`).
