# Миграция избранного (favoritesMigration)

Переносит локальный список избранных точек SVP/CUI в нативные «звёздочки» или «замочки» SBG 0.6.1.

## Возможности для игрока

В настройках модуля доступны:

- **⬆️ Импорт из JSON** — загружает локальный список SVP/CUI-избранных из ранее экспортированного JSON. **REPLACE-семантика**: текущий список в IDB полностью заменяется. Используется при переносе на новое устройство или восстановлении после wipe. После импорта флаг `lock-migration-done` сбрасывается — свежие легаси-точки требуют повторной миграции в нативные замочки.
- **⬇️ Скачать JSON** — выгружает текущий список в файл `svp-favorites-YYYY-MM-DD.json`. Формат — массив GUID-строк, отсортированный.
- **Перенести старый список в избранное (звезда)** — для каждой точки из локального списка SVP/CUI с ключами в инвентаре отправляется `POST /api/marks { guid, flag: 'favorite' }`. Это нативная «звёздочка» SBG 0.6.1: визуальное маркирование, не защищает от удаления автоочисткой.
- **Перенести старый список в заблокированное (замок)** — то же, но с `flag: 'locked'`. Нативный замочек: ключи не участвуют в рисовании и не удаляются автоочисткой `inventoryCleanup`. После успешной миграции выставляется флаг `svp_lock_migration_done`, и `inventoryCleanup` перестаёт блокировать удаление ключей по факту непустого legacy-списка.

После миграции локальный список SVP/CUI **не очищается**: миграцию можно повторить при необходимости (например, после случайного снятия флага в инвентаре игры).

## Поведение

### Per-stack обработка

Игра в SBG 0.6.1 хранит ключи как стопки (`item.t === 3`, `item.l === pointGuid`, `item.g === stackGuid`). У одной точки может быть несколько стопок инвентаря. Миграция вызывает `POST /api/marks` для **каждой стопки** избранной точки — флаг применяется на уровне стопки.

В UI инвентаря игры стопки агрегируются в одну строку, поэтому пользователь видит «звёздочка стоит у этой точки» независимо от того, на скольких стопках она физически отмечена.

### Sequential + rate-limit

Запросы идут строго последовательно (одна стопка за раз) с задержкой `DEFAULT_REQUEST_DELAY_MS = 1500` мс между ними. Эмпирически проверено: при `concurrency = 4` без задержки большая часть запросов возвращала `result: false` (отказ замаскированный под toggle-off); 30 sequential-запросов с интервалом 1500 мс прошли с 100% успехом. Миграция 100 точек занимает около 2,5 минут, но все стопки получат флаг.

### Retry-механизм

Запросы `POST /api/marks` могут провалиться двумя способами. Оба обрабатываются автоматически, без confirm-диалогов: пользователь нажал кнопку и просто ждёт результат.

1. **Toggle off** (`HTTP 200`, `result === false`) — сервер toggle'нул флаг OFF, наш предварительный фильтр не сработал из-за устаревшего `f` в кэше или лёгкой рассинхронизации client/server. После паузы `DEFAULT_TOGGLE_RETRY_DELAY_MS = 3000` мс упавшие стопки прогоняются ещё раз одним батчем. Если и второй раз `false` — стопка идёт в `toggleStuck` итогового результата.
2. **Сетевая ошибка** (fetch reject, HTTP не 2xx) — экспоненциальный backoff `DEFAULT_NETWORK_RETRY_DELAYS_MS = [5000, 15000, 45000]` мс. До трёх автоматических попыток на стопку; если ошибка остаётся после третьего retry, стопка попадает в `networkFailed` итогового результата.

Каждая фаза (`initial` / `retry-toggle` / `retry-network`) перезапускает прогресс-бар с `0/N` через колбэк `onPhaseChange` — пользователь видит чистый счётчик retry без скачка `total` с N до N+M.

## Хранилище

Source данных миграции — IndexedDB `CUI/favorites` (тот же store, что у CUI). На init модуль вызывает `loadFavorites()` из `core/favoritesStore`, который заполняет in-memory `Set<string>` GUID'ов точек.

После переименования из `favoritedPoints` в `favoritesMigration` точечные write-методы (addFavorite, removeFavorite) удалены: модуль не редактирует список по-точкам, нативные звёздочки/замочки игра пишет в свой формат `inventory-cache[i].f`. REPLACE-методы `exportFavoritesToJson` / `importFavoritesFromJson` сохранены в `core/favoritesStore` — нужны для переноса легаси-списка между устройствами и восстановления после wipe.

### Count seal (защита от потери IDB)

Сохранён из старой реализации: если IDB пуста, а localStorage `svp_favorites_seal > 0` — значит, данные потеряны (Android IDB wipe). `isFavoritesSnapshotReady()` возвращает false, `inventoryCleanup` не удаляет ключи, пользователь получает alert.

## Эндпоинт `POST /api/marks`

Payload `{ guid: string, flag: 'favorite' | 'locked' }`. Ответ — `{ result: boolean }` напрямую, без обёртки `response`: `result === true` означает, что флаг УСТАНОВЛЕН после toggle, `false` — снят. Формат подтверждён ручным fetch'ем в DevTools (release-notes игры приводили иной формат, фактический оказался без обёртки).

Игровой `apiSend` — IIFE-внутренняя функция и недоступна юзерскрипту. Используется прямой `fetch` с auth-токеном из `localStorage['auth']` (паттерн из `inventoryApi.deleteInventoryItems`).
