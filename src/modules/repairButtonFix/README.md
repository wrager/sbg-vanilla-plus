# repairButtonFix

Возвращает кнопку «Починить» из залипшего `disabled`-состояния.

## Симптом

После быстрого заряда точек под атакой (когда враг ломает, ты успеваешь зарядить) кнопка `#repair` остаётся серой и не нажимается до перезагрузки страницы. Происходит не каждый раз, на одной и той же серии действий по-разному.

## Корневая причина (предположение)

Игровой обработчик `#repair` ([refs/game/script.js:946-953](../../../refs/game/script.js#L946-L953)):

```js
$('#repair').on('click', async () => {
  $('#repair').addClass('locked').prop('disabled', true)
  const { response } = await apiSend('repair', 'post', {...})
    .catch(({ toast }) => apiCatch(toast, true))  // деструктуризация
  $('#repair').removeClass('locked').prop('disabled', false)
  ...
})
```

При rejection из `apiSend`, у которого нет поля `toast` (network abort при быстрой смене точки, AbortError при cancel предыдущего запроса, TypeError из чужого скрипта), деструктуризация `{ toast }` бросает `TypeError: Cannot destructure property 'toast' of undefined`. Строка снятия `locked`/`disabled` не выполняется - кнопка остаётся в исходном состоянии.

Прямого подтверждения через инструментацию не было: симптом редкий, воспроизводится только в живой битве на точке. Гипотеза основана на чтении кода и совпадает с описанным поведением (залипание, не каждый клик, требует reload).

## Защита

MutationObserver на классы атрибутов `#repair`. Когда добавлен класс `locked` (игра начала запрос), запускается таймер на 10 секунд. Если за это время игра штатно сняла `locked` (запрос завершился) - таймер отменяется. Если класс висит дольше - модуль снимает его и `disabled` атрибут.

10 секунд - запас: типичный repair-запрос отрабатывает <1 секунды, даже на медленной сети редко тянется больше 5. Если попадём в честный долгий запрос - повторный клик пользователя пошлёт второй repair, серверная сторона сама дросселирует параллельные.

## Чего НЕ покрывает

- Залипание по другим причинам (не через `locked`-класс): `disabled` атрибут отдельно от класса.
- Залипание в первые 10 секунд: пользователь сначала видит лаг, потом восстановление. Альтернатива - агрессивный таймаут - чревата ложными срабатываниями на медленной сети.
