# Переход к следующей точке (nextPointNavigation)

Кнопка в попапе точки - переход к ближайшей не посещённой точке в радиусе действия игрока (45 м). Дополняет нативный свайп: модуль ходит только по точкам в радиусе действия, нативный свайп - по всем точкам в зоне видимости карты.

## Возможности для игрока

- **Кнопка `>` в `.info .i-buttons`.** Появляется в попапе любой точки. Активна, когда в зоне действия игрока есть хотя бы одна не посещённая в текущей сессии точка. Disabled - когда таких точек нет (все посещены или их нет в радиусе).
- **Авто-зум до 17.** Если текущий зум меньше 16, после клика карта автоматически зумится до 17. Цель - сделать переход между точками визуально читаемым на любом масштабе.
- **Приоритет по полезности.** Из множества кандидатов в радиусе модуль выбирает в порядке (1) точки со свободными слотами для деплоя, (2) доступные для изучения, (3) любые. Внутри каждой группы - ближайшая по расстоянию.
- **Per-session трекинг посещений.** Точка считается "посещённой", когда её попап был открыт в текущей сессии. После reload счётчик обнуляется. `Set<guid>` хранится в памяти модуля.

## Различие с нативным свайпом

Нативный свайп между соседними точками (refs/game/script.js:722-752) появился в SBG 0.6.1. Он:

- Циклически перебирает массив `near_points` (все точки в зоне видимости карты, в порядке как сервер отдал).
- Привязан к Hammer.Swipe на `.info`: swipeleft - next, swiperight - prev.
- Не учитывает радиус действия игрока и историю посещений.

Наш модуль решает другую задачу: целенаправленный обход в зоне действия. Триггеры разные (свайп vs кнопка), критерии выбора разные. Конфликта между ними нет - пользователь получает оба варианта.

## Архитектура

### Player tracking

На enable модуль находит OL-слой `player` и подписывается на `change` его VectorSource. На каждый event (player двигается) функция `updateButtonStates` пересчитывает состояние кнопки в открытом попапе: есть ли в радиусе INTERACTION_RANGE = 45 м не посещённая точка.

Player не всегда есть в DOM на момент enable (может быть в загрузке). Подписка идёт через `change`-event source, который сработает когда первая точка добавится. До этого кнопка остаётся disabled.

### Popup observer

`MutationObserver` на `.info` с фильтром по `class` атрибуту - реагирует на открытие/закрытие попапа. Когда попап открывается, кнопка инжектируется в `.i-buttons`, состояние пересчитывается. Когда закрывается - кнопка не удаляется (popup сам прячется), но state остаётся.

### Авто-зум

После клика на нашу кнопку:

1. Извлекаем guid целевой точки.
2. Если `view.zoom < AUTOZOOM_THRESHOLD = 16`, ставим `view.zoom = AUTOZOOM_TARGET = 17`. Параллельно ставим флаг `autozoomInProgress = true`.
3. Симулируем клик по соответствующей feature (нативный handler откроет попап). Если symulация не сработала с первого раза (точка ещё не отрендерена при новом зуме), retry до `MAX_FAKE_CLICK_RETRIES = 3` раз с задержкой.
4. После открытия попапа `autozoomInProgress` сбрасывается.

### Geodesic distance

`getGeodeticDistance` использует `ol.sphere.getLength` на LineString из двух точек. Тот же метод, что игра в `isInRange`/`getDistance` (refs/game/script.js:2751-2757) - результат идентичен нативному критерию "точка в радиусе".

Внутри радиуса для упорядочивания по близости используется простое проецированное расстояние (`dx*dx + dy*dy`) - быстро и достаточно для сортировки в малом radius'е.

## Настройки

Модуль не имеет настроек. Включён по умолчанию. Не работает на устройствах без OL `sphere.getLength` (теоретически возможно при минимизированной сборке OL, в реальной игре всегда доступен) - в этом случае все расстояния возвращаются как Infinity и кнопка всегда disabled.
