# Переход к следующей точке (nextPointNavigation)

Свайп влево/вправо на попапе точки переходит к следующей точке в радиусе взаимодействия игрока (45 м) с приоритетом по полезности. Заменяет нативный горизонтальный свайп игры (который циклически перебирал все видимые точки в порядке как сервер отдал) на тот же жест с нашей навигацией.

## Возможности для игрока

- **Свайп влево/вправо на `.info`.** Открывает следующую точку. Оба направления зовут одну и ту же приоритетную навигацию (в отличие от нативного, где влево = next, вправо = prev). Свайп внутри карусели ядер (`.splide`) игнорируется - там работают нативные жесты карусели.
- **Анимация 120мс.** Попап улетает в направлении свайпа с fade-out, новый попап приходит на месте старого. Ощущение мгновенного перехода. Если следующей точки нет - попап откатывается на исходную позицию (return-анимация).
- **Приоритет по полезности.** Из множества кандидатов в радиусе модуль выбирает в порядке (1) точки со свободными слотами для деплоя, (2) доступные для изучения, (3) любые. Внутри каждой группы - ближайшая по расстоянию.
- **Per-session трекинг посещений.** Точка считается "посещённой", когда её попап был открыт в текущей сессии. После reload счётчик обнуляется. Когда все точки в радиусе посещены - цикл начинается заново.
- **Авто-зум до 17 при низком масштабе.** Когда попап открывается и в радиусе нет других точек, при `zoom < 16` модуль кратковременно зумится на игрока: игра подгружает точки в новой viewport-области, и следующий свайп их находит. После загрузки viewport восстанавливается.

## Различие с нативным поведением

Нативный горизонтальный свайп между соседними точками (refs/game/script.js:722-752) появился в SBG 0.6.1. Он:

- Циклически перебирает массив `near_points` (все точки в зоне видимости карты, в порядке как сервер отдал).
- Привязан к Hammer.Swipe на `.info`: swipeleft - next, swiperight - prev.
- Не учитывает радиус действия игрока и историю посещений.

Наш модуль решает другую задачу: целенаправленный обход в зоне действия. Нативный свайп подавляется через text-патч `gameScriptPatcher` (Hammer.Manager не создаётся), наш handler регистрируется в `core/popupSwipe` на `left`/`right` и обрабатывает жест по нашей логике.

## Архитектура

### Подавление нативного свайпа

`src/core/gameScriptPatcher.ts` при загрузке игры перехватывает основной script.js и применяет text-замену: блок `new Hammer(document.querySelector('.info'), { recognizers: [...] })` заменяется на comma-expression `(window.__svpNativeInfoSwipeDisabled = true, { on() {} })`. После замены `hammer_info` становится no-op объектом, `hammer_info.on('swipeleft swiperight', ...)` вешает callback в пустоту, нативные touch-listener'ы на `.info` не привязываются. Sentinel-флаг `window.__svpNativeInfoSwipeDisabled` доступен для диагностики.

Альтернатива через runtime-обёртку `Hammer.Manager.prototype.emit` (как было в раннем `improvedNextPointSwipe`) отвергнута в пользу text-патча: проектное правило приоритета (text-patch > runtime-обёртка), нет необходимости в monkey-patch на глобальный прототип, чище для других потенциальных потребителей Hammer.

### Свайп-handler

Модуль регистрирует `left` и `right` через `registerDirection` в `core/popupSwipe` с одним общим handler:

- `canStart(event)` - повторяет логику нативного фильтра: исключает touch внутри карусели слайдов (`.splide`), чтобы свайп ядер открывал/двигал слайдер, а не точку.
- `decide()` - вызывает `pickNextInRange`. Если есть следующая точка - помещает её guid в `pendingNextGuid` и возвращает `'dismiss'`. Если нет - возвращает `'return'`, попап откатывается без действий.
- `finalize()` - вызывается после dismiss-анимации (transitionend). Зовёт `openPointPopup(pendingNextGuid)` -> `window.showInfo`.
- `animationDurationMs: 120` - короче дефолтных 300мс из core: горизонтальный жест должен ощущаться мгновенно. 300мс маскировали бы apiQuery внутри showInfo, но цена - ощущение лага. 120мс - резкий отклик; кратко видна "дыра" между улетающим попапом и приходом нового на медленной сети, но воспринимается как загрузка, не как лаг скрипта.

State machine в `core/popupSwipe` (`idle -> tracking -> swiping -> animating -> idle`) сериализует жесты, поэтому `pendingNextGuid` не подвержен race conditions между параллельными свайпами.

### Открытие следующей точки

`openPointPopup(guid)` пробует `window.showInfo(guid)` напрямую (доступен после `gameScriptPatcher` экспозиции). Попап НЕ скрывается перед вызовом - showInfo сам зовёт `removeClass('hidden')` и обновляет содержимое (`refs/game/script.js:2120`); скрытие давало бы визуальное мерцание в течение await apiQuery. Нативный свайп игры (refs/game/script.js:751) тоже не скрывает попап.

При отсутствии `window.showInfo` (если `gameScriptPatcher` не сработал) - fallback на fake-click через карту: вычисляем pixel из координат feature и зовём `map.dispatchEvent({ type: 'click', pixel, ... })`. На fake-click есть retry-логика: если popup observer видит, что data-guid не сменился (промахнулись по соседней точке), повторяем до `MAX_FAKE_CLICK_RETRIES = 3` раз; если попали в другую точку - принимаем её как часть цепочки visited.

### Popup observer

`MutationObserver` на `.info` с фильтром по `class`/`data-guid` атрибутам и childList. Реагирует на:

- Открытие попапа (переход hidden -> visible) - сбрасывает retry-счётчики, проверяет в радиусе ли точка, если радиус пуст и зум низкий - запускает autozoom.
- Смену data-guid - либо это наш expectedNextGuid (нормальная цепочка), либо ручное открытие пользователем (сбрасывает visited). Логика отличает по `expectedNextGuid !== null`.

### Геодезическое расстояние

`getGeodeticDistance` использует `ol.sphere.getLength` на LineString из двух точек - тот же метод, что игра в `isInRange`/`getDistance` (refs/game/script.js:2751-2757). Результат идентичен нативному критерию "точка в радиусе". Внутри радиуса для упорядочивания по близости используется простое проецированное расстояние (`dx*dx + dy*dy`) - быстро и достаточно для сортировки в малом радиусе.

### Race-disable защита

`enable` содержит `await getOlMap()`. Если `disable` вызовется между стартом enable и резолвом промиса, текущая реализация без защиты записала бы map/pointsSource и зарегистрировала direction-handler'ы для логически-disabled модуля. Защита через `installGeneration` counter (тот же паттерн, что и в `pointTextFix`/`popoverCloser`/`nativeGarbageGuard`): enable увеличивает счётчик и запоминает свой generation; disable увеличивает его независимо. После await enable сверяет myGeneration с текущим installGeneration - если расходятся, выходим до registerDirection.

## Настройки

Модуль не имеет настроек. Включён по умолчанию. Не работает на устройствах без OL `sphere.getLength` (теоретически возможно при минимизированной сборке OL, в реальной игре всегда доступен) - в этом случае все расстояния возвращаются как Infinity и свайп не находит следующую точку.
